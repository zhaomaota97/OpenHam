# OpenHam 长期开发与能力版图清单 (Feature Backlog)

这份清单记录了当前 OpenHam 项目为达到成熟的自动化 AI 助手平台所建议补充的核心能力。我们可以先将其作为愿景图景记录在此，后续研发可根据实际需求优先级随时择机开展。

## 1. 核心痛点：AI 自动化与脚本赋能
- [x] **基于自然语言的 AI 脚本一键生成** 🌟 (已完成)
- [x] **零依赖便携式发布方案 (Portable Deployment)** 🌟 (已支持)
  - **背景**：终端用户往往没有 Python 环境配置能力，而本项目的 AI 脚本生态高度依赖 Python。
  - **成果**：在底层 `script_manager` 中完成了对项目局部独立环境（`runtime/python.exe`）的静默劫持识别。后续发版只需一并打包一个精简免安装的 Python 目录，终端用户即可“无感知”地双击免安装运行，完美摆脱系统依赖地狱。
- [ ] **草稿与 AI 生成历史记录本 (History & Versioning)**
  - **背景**：在多次使用 AI 尝试生成、或者大量测试临时运行后，偶尔希望找回某个曾经表现良好的旧版逻辑代码。
  - **设计**：取代目前底层无序堆积的 `script_uuid.py` 垃圾文件，建立正式的有时间戳的“本地草稿区/历史生成流”面板。用户随时可以按时间线穿梭召回旧版本代码。
- [ ] **复杂业务编排系统 (Workflow Orchestration)**
  - **背景**：脚本往往不是孤立的。
  - **设计**：支持拖拽式或节点式的流程图编排，上一条脚本的输出可以无缝作为下一条脚本/AI任务的输入边界。
- [ ] **被动触发与定时调度任务**
  - **背景**：不仅仅局限于语音唤起或点击唤起运行。
  - **设计**：引入 Cronjob 定时任务触发，以及基于文件改动、屏幕内容匹配等系统级事件的触发模式。

## 2. 架构基石：插件基座能力体系 (Plugin Core)
- [ ] **标准化插件生命周期与热插拔** 🌟
  - **背景**：目前已有插件概念，需进一步夯实基座统一标准。
  - **设计**：提供标准化的生命周期 API（`on_mount`, `on_unmount`, `on_message`）；支持无需重启项目即插即用的沙盒式插件热重载。
- [ ] **跨插件全局消息总线 (Event Bus)**
  - **背景**：解决多模块协同死局。
  - **设计**：全局暴露类似 Pub/Sub 的消息总线（例如语音组件派发 `audio_received`，脚本组件监听到后立即执行等），让不同基建插件无缝对话。
- [ ] **虚拟沙箱与权限阻隔**
  - **设计**：确保运行未知第三方插件或高风险生成的执行脚本不会脱离可控范围，提供权限审查或安全模式执行能力。

## 3. UI 视觉交互与展现层
- [ ] **系统托盘功能增强与样式调优 (System Tray V2.0)** 🌟
  - **设计**：抛弃简陋呆板的原生 Context Menu，像现主流辅助软件（如 Clash 托盘面板）一样，鼠标悬浮或点击后直接拉起充满高级视觉设计感的弹窗菜单，提供常用脚本的一键启停进度条展示，甚至微型问答输入框。
- [ ] **沉浸式控制台视觉交互动画**
  - **设计**：针对执行中的任务或 Boss 仿真器，补充类似于呼吸灯、进度推移和打字机光晕的高级过渡动画反馈，拉升客户端产品的体感。

## 4. 全局可观测与生态扩展
- [ ] **全局统计仪表盘与日志画像 (Telemetry)**
  - **设计**：记录各种工具被调用的频次地图，帮助用户发现效率瓶颈；统计消耗大模型 Token 额度，呈现脚本运行的历史错误率监控台。
- [ ] **云端配置与工作区互通同步**
  - **设计**：实现所有 Trigger（配置与运行脚本）支持一键向内导出导入（Zip / JSON组合），未来甚至整合 Gitee / GitHub 同步机制，做云端漫游方案。
- [ ] **自建云端社区市场 (Plugin/Script Registry)**
  - **设计**：支持从类似于 “OpenHam 公开集市” URL 一键订阅和下载别人写好的复杂优秀脚本与预制插件，完善社区生态雏形。
